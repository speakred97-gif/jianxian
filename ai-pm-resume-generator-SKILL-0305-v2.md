# ai-pm-resume-generator-SKILL.md
**version:** 3.0  
**适用场景：** 将AI产品经理的项目文档（PRD/项目复盘/自由描述）转化为专业简历  
**支持模式：** 标准模式（文档质量好）/ 增强模式（文档质量弱）

---

## 使用本Skill前的必读规则

### 输出规则 ⚠️ 最高优先级

1. **始终直接输出简历**，无论文档质量好坏，诊断在内部执行，不打断用户等待确认
2. **简历bullet禁止写功能描述句**，每条bullet必须有：主语（我）+ 动作（做了什么）+ 结果（带来什么）
3. **禁止在简历正文出现「三层成果表格」**，成果数字内嵌进每条bullet末尾
4. **禁止写入简历的表达：**
   - 「梳理业务流程」「产出PRD/原型图」「参与需求评审」——无法体现AI技术价值
   - 「综合能力强」「同理心」「深入用户」——虚词，无法量化
   - 成果没有数字的条目——必须有至少一个量化指标，实在没有用【待核实】标注
5. **每条bullet强制执行因果链结构：**
   > 针对[发现的问题/痛点] → [设计/主导/优化了什么] → [带来什么量化结果]

## 用户需要提供的输入

### 必填项（缺少任一项必须阻断流程，提示用户补充）

| 输入 | 必填 | 说明 |
|------|------|------|
| **姓名** | ✅ 必填 | 可用化名，不影响生成质量 |
| **工作经历** | ✅ 必填 | 公司名、部门、职位、时间，提取到简历里 |
| **项目文档** | ✅ 必填 | 至少一个项目，上传文件或粘贴文字均可 |

### 建议填写项

| 输入 | 说明 |
|------|------|
| **目标职位** | 不填默认「AI产品经理」 |
| **教育背景** | 不填则简历缺教育模块 |

### 可选项

| 输入 | 说明 |
|------|------|
| **联系方式** | 电话 / 微信 / 邮箱，不填留空 |
| **工作年限** | 帮助定位 senior / junior 级别 |
| **补充信息** | 证书、技术栈、SCI论文等加分项 |

### 多项目文档支持

**每个项目单独输入，独立诊断，合并输出到同一份简历。**

- 每个项目可以上传多个文件（PRD、复盘文档、需求文档等），也可以直接粘贴文字
- 每个项目独立走「类型识别 → 模块选配 → 诊断 → 内容提取」全流程
- 多个项目按工作经历时间顺序合并输出到简历的「项目经验」模块
- 每个项目可以命名（如「智能外呼机器人」），诊断完成后自动标注识别出的类型标签

**多项目处理顺序：**
1. 逐个项目跑诊断（并行显示进度）
2. 所有项目诊断完成后，一次性调用简历生成（上下文包含所有项目）
3. 生成一份完整简历，每个项目作为独立项目块

## 第一步：项目类型识别

拿到项目文档后，**首先判断项目类型**，这决定了后续选配哪些工作模块。

| 类型代码 | 项目类型 | 典型产品举例 |
|----------|----------|-------------|
| `LLM-RAG` | 检索增强生成 | 企业知识库问答、法律助手、客服机器人、政务问策 |
| `LLM-GEN` | 纯生成类 | AI写作、公文生成、营销文案、代码生成 |
| `LLM-AGENT` | 智能体/工作流自动化 | 智能外呼机器人、AI RPA、多步骤任务执行 |
| `LLM-CONV` | 对话/多轮交互 | 智能客服、陪伴机器人、AI面试官 |
| `CV` | 计算机视觉 | 图像识别/分类、目标检测、OCR、医疗影像 |
| `CV-GEN` | 视觉生成 | AI生图、图像编辑、虚拟试衣、数字人 |
| `ASR/TTS` | 语音识别/合成 | 语音转写、会议纪要、语音克隆、TTS播报 |
| `MULTIMODAL` | 多模态融合 | 图文理解、视频问答、多模态内容审核 |
| `REC` | 推荐/排序系统 | 内容推荐、商品推荐、个性化Feed、广告CTR |
| `CLASSIFY` | 分类/判别类 | 文本分类、情感分析、意图识别、风控评分 |
| `FINETUNE` | 模型微调/训练 | 垂直领域大模型、行业精调模型、SFT/RLHF |
| `DATA` | 数据智能/分析 | AI数据分析平台、智能BI、数据标注平台 |

**一个项目可以同时属于多个类型**，例如：
- 智能外呼机器人 = `LLM-AGENT` + `ASR/TTS` + `CLASSIFY` + `FINETUNE`
- 公文写作智能体 = `LLM-GEN` + `LLM-RAG` + `LLM-CONV`

## 第二步：文档质量诊断（内部执行，不展示给用户）

按选配模块评估文档完整度，给出 0-100 分：
- **≥ 70 分**（5个以上模块有实质内容）→ 标准模式，直接生成简历
- **< 70 分**（3个以上模块缺失或无数字）→ 增强模式，AI补全后生成简历，补全内容加 `*` 标注

---

## 第三步：动态选配工作模块

### 选配规则

```
必选（所有项目）：U1 + U2 + U3 + U4

按类型追加：
  含 LLM-RAG    → 追加 R1 + R2
  含 LLM-GEN    → 追加 G1 + G2 + G3（若有引导交互）
  含 LLM-AGENT  → 追加 A1 + A2 + A3
  含 CV         → 追加 C1 + C2
  含 CV-GEN     → 追加 C3
  含 ASR/TTS    → 追加 S1（+ S2 若有声音定制）
  含 REC        → 追加 RC1 + RC2
  含 FINETUNE   → 追加 F1 + F2
  含 LLM-CONV   → 追加 G2 + A2 + G3
  含 CLASSIFY   → 追加 A2
```

## 模块池（工作维度库）

### 通用模块（所有项目必选）

---

**U1｜需求调研与场景定义**

写法公式：
> 针对[用户群体]在[场景]下[痛点描述]的问题，调研[N名用户/N份问卷]，识别[X类]高频场景，完成[N轮]需求迭代

简历示例：
> 针对一线销售人员外呼话术无法匹配用户行业的问题，调研阿里、腾讯等4家竞品及一线销售流程，识别存量激活、活动邀约等5类高频场景，完成3轮需求验证迭代

---

**U2｜大模型/算法选型**

写法公式：
> 针对[场景对模型的特殊要求]，基于[N个评测维度]构建[N条]评测集，横向对比[X个]模型/方案，选定[最终方案]，在[关键指标]上表现最优

LLM类关注：幻觉率、指令遵循、中文能力、私有化部署  
CV类关注：mAP、推理速度、部署成本  
推荐类关注：AUC、召回率

---

**U3｜评测体系建设**

写法公式：
> 针对[缺乏标准化评估导致XX问题]，设计[N个维度]评测标准，构建[X条]测试用例集，建立[人工+自动]双轨评测机制，支撑模型[周/双周]迭代闭环

---

**U4｜Badcase迭代闭环**

写法公式：
> 建立「[触发] → 问题分类 → 归因分析 → [修复方式] → 回归测试」[周/双周]闭环，累计处理[N类]问题，完成[X轮]版本迭代，[核心指标]提升[X%]

## 第四步：业绩成果处理规则

**三层成果的正确用法：**
- 作为内部思考框架，确保每个项目都覆盖了三层价值
- **不作为简历中的独立模块或表格**，数字全部内嵌进对应bullet末尾

**三层定义：**

| 层级 | 内容 | 对应指标举例 |
|------|------|-------------|
| 第一层：模型/技术效果 | AI本身跑出来的指标 | 准确率/召回率/mAP/MOS/质量评分/延迟 |
| 第二层：业务价值 | 对内部效率/用户体验的提升 | 人效提升X%/耗时缩短X%/处理量增加X倍 |
| 第三层：商业价值 | 对公司收入/客户/市场的贡献 | 合同金额/客户数/续费率/成为标杆案例 |

---

## 第五步：AI补全规则（增强模式）

当文档完整度 < 70 分时启用：

1. **原文已有的内容**：原样整理，不修改实质内容
2. **AI基于行业常识推断的内容**：
   - 简历中在对应条目末尾加 `*` 号
   - 数字后标注行业参考区间，例如：`意图识别准确率达92%（行业参考：88%～96%）*`
3. **简历底部必须附说明：**

```
* 内容来源说明
简历中标注 * 号的内容，部分来自AI基于行业常识的推断补全，
并非来自原始项目文档记录。建议使用前与真实情况核对并修改。
```

4. **AI补全不允许的行为：**
   - 捏造与原文明显矛盾的内容
   - 将AI推断内容洗白为无标注的普通文字
   - 补全独特的个人故事（如「用GPT-5生成测试集」这类细节只有本人知道）

（结合 /mnt/skills/public/docx/SKILL.md 使用，用 docx-js 生成）

- **配色：**
  - 姓名/页眉：深蓝 `#1F3864`

- **列表缩进：** left:480 DXA，hanging:280 DXA
- **行距：** 276 twips
- **禁止：** 用表格做分割线或布局（打印显示为空白框）
- **bullet：** 使用 LevelFormat.BULLET，不得用 unicode 字符「•」

## 常见Q&A

**Q：用户没有具体数字怎么办？**  
A：增强模式下AI给出行业参考区间，用`*`标注；如用户能补充则追问一次，用户无法补充则保留参考区间占位。

**Q：项目文档只有一个项目怎么办？**  
A：基于该项目生成，不虚构第二个项目，可在「工作经历」部分适当扩充其他非AI工作内容。

**Q：用户不想透露真实姓名怎么办？**  
A：接受化名或「XX」占位，提醒用户最后自行填写。

**Q：如何判断某个模块是否适用？**  
A：严格按照项目类型代码选配，不因为「大部分AI项目都有RAG」就默认加入，必须文档中有相关技术描述才选配对应模块。

**Q：原始文档里有些数字，AI补全了更好的数字怎么处理？**  
A：绝对不允许。原文有数字必须用原文数字，AI补全只能填补原文完全缺失的部分。
